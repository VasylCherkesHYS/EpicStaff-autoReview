Telegram Trigger Node Overview
==============================

The Telegram Trigger feature enables you to start graph execution workflows from
events generated by a Telegram bot. This document explains the data model and
services involved.

1. Data Model
-------------

The following database models persist configuration and mapping logic:

- **TelegramTriggerNode**  
  - Represents the entry point within a graph.  
  - Stores the `telegram_bot_api_key`.  
  - Linked via a `webhook_trigger` foreign key to a centralized
    `WebhookTrigger` entity.  
  - **Note**: The webhook path (UUID) is now managed by the parent `Webhook`
    entity, not directly on the node.

- **NgrokWebhookConfig**  
  - Stores tunneling details (domain, auth token, region, etc.) for the
    webhook.  
  - Determines which public URL will be used to receive Telegram updates.

- **TelegramTriggerNodeField**  
  - Defines the data mapping strategy between the incoming Telegram payload
    and the graph state.  
  - Each record links a specific payload field (for example, `message.text`)
    to a target variable path in the graph (for example,
    `variables.telegram_data.user_input`).

2. Service Layer
----------------

The `TelegramTriggerService` coordinates registration and event handling and
works closely with the `WebhookTriggerService` for tunnel management.

### 2.1 Registration Flow

The `register_telegram_trigger` method:

1. Accepts a `TelegramTriggerNode` instance.
2. Validates that the node has an associated `webhook_trigger` and a valid
   `NgrokWebhookConfig`.
3. Uses `WebhookTriggerService().get_tunnel_url()` to resolve the active
   public tunnel URL.
4. Calls the Telegram Bot API `setWebhook` endpoint to register the webhook
   URL for the bot associated with `telegram_bot_api_key`.

### 2.2 Event Handling Flow

The `handle_telegram_trigger` method:

1. Receives raw webhook payloads on specific paths, managed by the
   centralized `WebhookTrigger`.
2. Finds the relevant `TelegramTriggerNode` instances by filtering on
   `webhook_trigger__path`.
3. Identifies the target graph and entrypoint node from the matching node(s).
4. Injects the incoming payload into the session, using
   `TelegramTriggerNodeField` mappings to populate graph variables.
